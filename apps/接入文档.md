# 游戏服务器接入文档

> **版本**: v1.0  
> **更新时间**: 2026-04-25  
> **适用对象**: 第三方游戏服务器开发团队

---

## 一、概述

本文档描述了游戏服务器需要实现的 HTTP 接口规范。平台后台通过这些接口与游戏服务器通信，实现玩家管理、邮件发送、充值到账等功能。

### 通信方式
- **协议**: HTTP/HTTPS
- **数据格式**: JSON
- **字符编码**: UTF-8
- **超时设置**: 一般接口 10 秒，支付通知 30 秒

### 基础地址

游戏服务器需要在后台管理系统中配置一个基础地址（`webhost`），如：  
`http://your-game-server:8888`

所有接口路径基于此地址拼接。

---

## 二、统一响应格式

所有接口必须返回以下 JSON 格式：

```json
{
  "code": 0,
  "message": "ok",
  "data": {}
}
```

| 字段 | 类型 | 说明 |
|---|---|---|
| `code` | number | 结果码。`0` = 成功，非 `0` = 失败 |
| `message` | string | 结果描述。成功时返回 `"ok"`，失败时返回错误描述 |
| `data` | object | 可选。业务返回数据 |

### 错误码规范

| code | 含义 |
|---|---|
| 0 | 成功 |
| 1001 | 参数错误 |
| 1002 | 玩家不存在 |
| 1003 | 服务器内部错误 |
| 1004 | 重复操作（幂等拦截） |
| 1005 | 权限不足 |

> 游戏方可扩展自定义错误码（建议从 2000 开始），平台侧只判断 `code === 0` 为成功。

## 四、接口详细说明

### 1. 封号 `/api/player/ban`

封禁指定玩家账号，禁止其登录游戏。

**请求**:
```http
POST /api/player/ban
Content-Type: application/json
```

```json
{
  "openId": "10001",
  "serverId": "1",
  "platform": "android",
  "duration": 86400,
  "reason": "违规操作"
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| openId | string | ✅ | 玩家唯一标识 |
| serverId | string | ✅ | 区服ID |
| platform | string | ✅ | 平台：`android` 或 `ios` |
| duration | number | ✅ | 封禁时长（秒）。`0` = 永久封禁 |
| reason | string | ✅ | 封禁原因 |

**成功响应**:
```json
{ "code": 0, "message": "ok" }
```

### 3. 发物资邮件 `/api/mail/send-with-items`

通过游戏内邮件向玩家发送物品。这是使用最频繁的接口，用于 GM 发放、CDK 兑换、礼包购买等场景。

**请求**:
```json
{
  "openId": "10001",
  "serverId": "1",
  "platform": "android",
  "roleId": "player_uuid_123",
  "serialNo": "1745000000_abc123def",
  "mailTitle": "系统奖励",
  "mailContent": "恭喜您获得奖励，请查收！",
  "items": [
    { "itemId": 1001, "itemCount": 100 },
    { "itemId": 2002, "itemCount": 5 }
  ]
}
```

| 字段 | 类型 | 必填 | 说明 |
|---|---|---|---|
| openId | string | ✅ | 玩家唯一标识 |
| serverId | string | ✅ | 区服ID |
| platform | string | ✅ | 平台：`android` 或 `ios` |
| roleId | string | ❌ | 角色ID（精确到角色时传入） |
| serialNo | string | ✅ | 幂等序列号（平台生成，用于防重复发放） |
| mailTitle | string | ✅ | 邮件标题 |
| mailContent | string | ✅ | 邮件正文 |
| items | array | ✅ | 物品列表 |
| items[].itemId | number | ✅ | 物品ID |
| items[].itemCount | number | ✅ | 物品数量（正整数） |

> **幂等性要求**: 游戏服务器应根据 `serialNo` 做去重，相同 `serialNo` 的请求只执行一次，重复请求直接返回成功。

**成功响应**:
```json
{ "code": 0, "message": "ok" }
```

## 五、通用字段说明

### `openId` — 玩家唯一标识

由平台分配给每个玩家账号（或子账号）的唯一标识符。游戏服务器需要维护 `openId` 与游戏内角色的对应关系。

### `serverId` — 区服ID

区服编号，在后台管理系统中配置。游戏服务器接收到请求后，应根据此字段定位到对应的区服进行操作。

### `platform` — 平台标识

| 值 | 说明 |
|---|---|
| `android` | 安卓平台 |
| `ios` | iOS 平台 |

### `serialNo` — 幂等序列号

平台侧自动生成的唯一字符串，格式为 `{时间戳}_{随机串}`，如 `1745568000_abc123def`。

**重要**: 游戏服务器**必须**根据 `serialNo` 实现幂等性：
- 首次收到某个 `serialNo` 时正常执行操作
- 再次收到相同 `serialNo` 时直接返回成功，**不重复执行**

## 六、请求签名验证（必须实现）

所有请求都携带签名，游戏服务器**必须验证签名**后才能执行业务逻辑。

### 签名方式

平台在每个请求的 HTTP Header 中携带以下三个字段：

| Header | 说明 | 示例 |
|---|---|---|
| `X-Timestamp` | 秒级 Unix 时间戳 | `1745568000` |
| `X-Nonce` | 随机字符串（防重放） | `m1abc_r4nd0mstr12` |
| `X-Sign` | HMAC-SHA256 签名 | `a1b2c3d4e5f6...`（64位十六进制） |

### 签名算法

```
签名载体 = timestamp + "\n" + nonce + "\n" + requestBody
签名结果 = HMAC-SHA256(签名载体, secretKey).toHex()
```

其中：
- `timestamp` = `X-Timestamp` 头的值（字符串）
- `nonce` = `X-Nonce` 头的值（字符串）
- `requestBody` = 完整的 HTTP 请求 Body（JSON 字符串，原样参与签名）
- `secretKey` = 双方约定的密钥（由平台分配，在后台配置）
- `\n` = 换行符（`0x0A`），不是字面的两个字符

### 验签步骤

```
1. 从 Header 中取出 X-Timestamp、X-Nonce、X-Sign
2. 检查 timestamp 与当前服务器时间的偏差不超过 60 秒（防重放）
3. 读取完整的请求 Body（原始字符串，不要反序列化后再序列化）
4. 拼接签名载体：timestamp + "\n" + nonce + "\n" + body
5. 用 secretKey 计算 HMAC-SHA256，得到 expectedSign
6. 比较 expectedSign 与 X-Sign 是否一致
```


```java
import javax.crypto.Mac;
import javax.crypto.spec.SecretKeySpec;

public static boolean verifySign(String timestamp, String nonce,
                                  String body, String sign, String secretKey) {
    long now = System.currentTimeMillis() / 1000;
    if (Math.abs(now - Long.parseLong(timestamp)) > 60) return false;

    String payload = timestamp + "\n" + nonce + "\n" + body;
    Mac mac = Mac.getInstance("HmacSHA256");
    mac.init(new SecretKeySpec(secretKey.getBytes("UTF-8"), "HmacSHA256"));
    byte[] hash = mac.doFinal(payload.getBytes("UTF-8"));

    StringBuilder sb = new StringBuilder();
    for (byte b : hash) sb.append(String.format("%02x", b));
    return sb.toString().equals(sign);
}
```

### 签名验证失败时

如果签名验证不通过，返回：
```json
{ "code": 1005, "message": "签名验证失败" }
```

### 安全须知

1. **secretKey 保密**：密钥不可硬编码在客户端代码中，建议通过环境变量或配置中心管理
2. **时间同步**：服务器时间需要与 NTP 同步，允许 ±60 秒偏差
3. **nonce 去重**（推荐）：记录 60 秒内收到的 nonce，拒绝重复 nonce
4. **IP 白名单**（推荐）：配合签名验证，限制仅平台 IP 可访问
5. **HTTPS**：生产环境建议使用 HTTPS 传输

## 七、对接流程

```
1. 游戏方按照本文档实现所有"必须"接口
2. 提供游戏服务器地址（webhost），如 http://1.2.3.4:8888
3. 在后台管理系统 → 服务器配置中添加服务器信息
4. 使用后台 GM 工具逐一测试每个接口
5. 确认所有接口正常后，正式上线
```

## 八、FAQ

**Q: 物品ID从哪里获取？**  
A: 物品ID由游戏方定义，需要提供一份物品配置表（包含 itemId 和物品名称），导入到后台管理系统中。

**Q: 是否支持多区服？**  
A: 支持。每个区服对应一条 GameServers 配置记录，包含独立的 webhost 地址。

**Q: openId 是什么？**  
A: 平台为每个用户（或子账号）分配的唯一ID。游戏首次收到某个 openId 时应自动创建对应的游戏角色。
